## PostgreSQL
- **The four PG19 features you'll actually feel in production** — Christophe Pettus skips the SQL/PGQ graph headline and flags the operational changes: 64-bit MultiXact members (kills the "vacuum-or-die" member-exhaustion failure mode), parallel autovacuum index workers (watch the `max_workers × max_parallel × maintenance_work_mem` worst case), temporal `UPDATE/DELETE ... FOR PORTION OF`, and `jit = off` by default (OLAP shops should re-enable and benchmark before upgrading). [link](https://thebuild.com/blog/2026/05/18/postgresql-19-beta-the-four-features-youll-actually-feel/)
- **pg_sorted_heap 0.14.0** — physically-sorted heap storage with zone-map pruning and a planner-integrated vector-search path. [link](https://www.postgresql.org/about/news/pg_sorted_heap-0140-released-3300/)
    - `sorted_hnsw` made planner-visible via the Index AM path for covered KNN, including parent Merge Append over leaf HNSW indexes
    - composite zone-map metadata (first-key block ranges + bounded second-key summary); `sorted_heap_bulk_load_ordered(...)` trusted ordered ingestion
    - verified on PostgreSQL 16, 17, 18; now on PGXN
- **pg_tre 1.1.1 — approximate-regex index access method** — a real `IndexAmRoutine` (USING tre) doing Levenshtein-distance regex matching ("within k edits of this pattern"), with WAL coverage, VACUUM awareness, and REINDEX CONCURRENTLY; three-tier filter funnel (BRIN-style range bloom → trigram postings → per-tuple bloom) over Laurikari's TRE library. Fills a gap pg_trgm/FTS/pgvector don't cover. PostgreSQL 18+. [link](https://www.postgresql.org/about/news/pg_tre-111-released-an-approximate-regex-index-am-for-postgresql-18-3305/)
- **pg_mentat 1.3.0 — Datomic-compatible Datalog inside Postgres** — immutable datoms, a Datalog compiler that lowers to ordinary SQL, the pull API, time-travel (as-of/since/history), and ACID via PG transactions; built with pgrx in Rust for PG13–18, with soft integrations into pg_trgm/fuzzystrmatch/pgvector/PostGIS/etc. [link](https://www.postgresql.org/about/news/pg_mentat-130-released-datomic-compatible-datalog-inside-postgresql-3306/)
- **plpgsql_wrap v1.0 — Oracle-style PL/pgSQL obfuscation** — stored procedures written `LANGUAGE plpgsql_wrap` are AES-256-GCM-encrypted into `pg_proc.prosrc`; source is unreadable (including in `pg_dump`, whose output stays restorable) without the key, otherwise behaves identically to plpgsql. PostgreSQL 12+. [link](https://www.postgresql.org/about/news/plpgsql_wrap-v10-released-3299/)
- **pg_statviz 1.0** — minimalist in-database stats time-series extension, now with an opt-in `--ai` analysis flag. [link](https://www.postgresql.org/about/news/pg_statviz-10-released-with-ai-powered-analysis-3301/)
    - `--ai` sends each chart's data/PNG to a vision LLM (Claude/Gemini/local Ollama) for a HEALTHY/WARNING/CRITICAL verdict plus remediation
    - notable design discipline: a deterministic rules engine runs first and a "severity floor" prevents the LLM downgrading a real finding; prompts wrap user data in `<user_data>` envelopes for prompt-injection containment and explicitly debunk shared_buffers/random_page_cost folklore
    - AI is fully optional; with no `--ai` flag there are zero extra dependencies and no external calls
- **check_pgactivity 2.10** — new release of the Nagios/monitoring plugin for PostgreSQL. [link](https://www.postgresql.org/about/news/release-check_pgactivity-210-3302/)

## Migration experience
- **21 CFR Part 11 audit logging in Postgres without triggers** — Harbor's Nathan Leung walks through why they rejected pgAudit (no app context) and PL/pgSQL triggers (`SET LOCAL` plumbing, per-study-DB duplication, PgBouncer transaction-pooling wariness) and instead wrapped their Drizzle ORM in a JS `Proxy` that forces `.withAuditLogCtx(...)` before any mutation, writing the data change and a signed before/after audit row in one transaction (`SELECT ... FOR UPDATE` for a consistent snapshot). Honest trade-offs section: extra round trips, weaker guarantee than a DB-level trigger, and a note on eventually moving signing into pgcrypto. [link](https://runharbor.com/blog/2026-05-18-building-automatic-contextual-clinical-trial-audit-logging-in-postgres-without-triggers)
